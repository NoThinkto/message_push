# Opportunity Pipeline 整体设计方案

## 1. 设计目标

本次新增的 `opportunity` 流程不是替换原有 `TrendRadar` 热点推送逻辑，而是在现有抓取、存储、通知渠道之上，增加一套独立的“机会识别与门控推送”业务流程。

设计目标如下：

1. 保留原有 `legacy` 流程的完整行为，不删除、不注释、不破坏。
2. 新增一套完全独立的上层业务目录，用于：
   - 机会识别专用提示词
   - 机会判断结构化结果
   - 推送门控
   - 机会预警文案渲染
3. 继续复用原有稳定的底层基础设施：
   - 数据抓取渠道
   - SQLite / 远程存储
   - 企业微信 / Telegram / 邮件等底层 sender
4. 通过配置项在上层选择使用旧流程还是新流程。

## 2. 目录边界

### 复用旧代码的部分

- 抓取：`trendradar/crawler/`
- 存储：`trendradar/storage/`
- 通知底层发送器：`trendradar/notification/senders.py`
- 配置加载主入口：`trendradar/core/loader.py`
- 主入口：`trendradar/__main__.py`

### 新增独立目录

`trendradar/opportunity_pipeline/`

这个目录只承载“机会识别流程”的业务逻辑，不复制底层 HTTP 推送、不复制爬虫、不复制数据库实现。

## 3. 流程说明

新流程的目标链路如下：

1. 复用现有数据抓取结果
2. 复用现有筛选结果（后续可再扩展为专属筛选）
3. 调用专属提示词进行 AI 机会识别
4. 解析固定 JSON 结果
5. 使用门控规则判断是否允许推送
6. 如果允许，则使用新的机会预警文案进行推送

## 4. 配置策略

在 `config/config.yaml` 中新增：

- `pipeline.mode`
  - `legacy`：使用旧流程
  - `opportunity`：尝试使用新流程

- `opportunity_pipeline`
  用于承载新流程的全部参数，例如：
  - 是否启用
  - 机会提示词文件
  - 机会流程专用兴趣文件
  - 置信度阈值
  - 允许的风险等级
  - AI 失败时的处理策略
  - 新流程允许使用的通知渠道

## 5. 当前骨架状态

本次提交先完成“结构骨架”和“配置骨架”，包括：

1. 新目录与数据结构
2. 新提示词文本
3. 新流程配置段
4. 主入口的流程模式识别与安全回退
5. 中文注释、中文文档、中文配置说明

当前不会替换旧流程，也不会强制启用新流程。即使把 `pipeline.mode` 改成 `opportunity`，在机会流程尚未完全实现前，程序也会输出清晰提示并自动回退到 `legacy`，避免打断现有使用。

4. 完成新流程专属通知分发
5. 最后接入完整执行链路
